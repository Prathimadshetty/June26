    SAST Tool and CI Integration

    At M&S Fortify On Demand(FOD) is being used for Static Application Security Testing (SAST).

    Fortify On Demand Analyzes the Source code for Application security vulnerabilities. Static Application Security Testing (SAST) can be integrated to CI/CD to trigger scan when a build process is initiated

    Integration steps:¶

    Azure Devops:¶

    Step 1: Store GitHub Username, Personal Access Token and API_KEY and API_SECRET in Azure Keyvault.

    Step 2: Create a Pipeline.yml with following code

    # Starter pipeline
​
# Start with a minimal pipeline that you can customize to build and deploy your code.
# Add steps that build, run tests, deploy, and more:
# https://aka.ms/yaml
​
trigger: none
​
jobs:
- job: FOD_Scanning
  pool:
    name: poc-2-test
  steps:
​
​
  - task: Bash@3
    inputs:
      targetType: 'inline'
      script: |
        wget -c https://github.com/fod-dev/fod-uploader-java/releases/download/v5.4.0/FodUpload.jar
        pwd
        ls -lrt

  - task: Bash@3
    inputs:
      targetType: 'inline'
      script: |
        apt install zip
        zip -r 'WebGoat-develop.zip' .

  - task: AzureKeyVault@1
    inputs:
      azureSubscription: 'AzureResourceManager'
      KeyVaultName: 'cicd-v2-shared-kv'
      SecretsFilter: '*'
      RunAsPreJob: false
  - task: Bash@3
    inputs:
      targetType: 'inline'
      script: |
        java -jar FodUpload.jar -ac $(FOD-API-KEY) $(FOD-API-SECRET) -rid $(RELEASE-ID) -purl $(FOD-PORTAL-URL) -apiurl $(FOD-API-URL) -z WebGoat-develop.zip -ep 2 -rp 0 -pp 1 -apf -I 1
    displayName: 'FOD SAST Scan'

  - task: Bash@3
    inputs:
      targetType: 'inline'
      script: |
             auth=$(curl -X POST --header 'Content-Type: application/x-www-form-urlencoded' --header 'Connection: close' -d 'grant_type=client_credentials&scope=api-tenant&client_id=$(FOD-API-KEY)&client_secret=$(FOD-API-SECRET)' 'https://api.emea.fortify.com/oauth/token')
             tok3n=$( echo "$auth" | jq '.["access_token"]' )
             tok3n="${tok3n:1:-1}"
             releasedetails=$(curl -X GET --header 'Accept: application/json' --header 'Authorization: Bearer '${tok3n} 'https://api.emea.fortify.com/api/v3/releases/'$(RELEASE-ID))
             scanid=$( echo "$releasedetails" | jq .currentStaticScanId )
             applicationname=$( echo "$releasedetails" | jq .applicationName | sed 's/["]//g')
             scanSummary=$(curl -X GET --header 'Accept: application/json' --header 'Authorization: Bearer '${tok3n} 'https://api.emea.fortify.com/api/v3/scans/'${scanid}'/summary')
             CRITICAL=$( echo "$releasedetails" | jq .staticCritical )
             HIGH=$( echo "$releasedetails" | jq .staticHigh )
             if [ $CRITICAL -gt 0 ]
             then
               echo "##[error]Application has critical Issues"
               curl -H 'Content-Type: application/json' -d '{"text": "Pipeline had warnings because critical issues in FoD SAST are present for '$applicationname'"}' $(teams_webhook)
               #exit 1
             elif [ $HIGH -gt 0 ]
             then
               echo "##[error]Application has High Issues"
               curl -H 'Content-Type: application/json' -d '{"text": "Pipeline had warnings because high issues in FoD SAST are present for '$applicationname'"}' $(teams_webhook)
               #exit 1
             else
               echo "PASSED"
             fi
    continueOnError: true           
    displayName: 'FOD Security Gating'

    FOD-PORTAL-URL - https://emea.fortify.com/

    FOD-API-URL - https://api.emea.fortify.com/

    Note: To enforce security gating in block mode please uncomment exit code 1.

    Github Actions¶

    Fortify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: DigitalInnovation/cloud-devsecops-pipelineactions/workflows/appsec/fortify@v1.1.1
        with:
          sast_release_id: ${{secrets.FOD_RELEASE_ID}}
          sast_api_key: ${{ secrets.FOD_API_KEY}}
          sast_api_secret: ${{ secrets.FOD_API_SECRET}}

      - name: FOD Security Gating
        run: |
          auth=$(curl -X POST --header 'Content-Type: application/x-www-form-urlencoded' --header 'Connection: close' -d 'grant_type=client_credentials&scope=api-tenant&client_id=${{secrets.FOD_API_KEY}}&client_secret=${{secrets.FOD_API_SECRET}}' 'https://api.emea.fortify.com/oauth/token')
          tok3n=$( echo "$auth" | jq '.["access_token"]' )
          tok3n="${tok3n:1:-1}"
          releasedetails=$(curl -X GET --header 'Accept: application/json' --header 'Authorization: Bearer '${tok3n} 'https://api.emea.fortify.com/api/v3/releases/'${{secrets.FOD_RELEASE_ID}})
          scanid=$( echo "$releasedetails" | jq .currentStaticScanId )
          applicationname=$( echo "$releasedetails" | jq .applicationName | sed 's/["]//g')
          scanSummary=$(curl -X GET --header 'Accept: application/json' --header 'Authorization: Bearer '${tok3n} 'https://api.emea.fortify.com/api/v3/scans/'${scanid}'/summary')
          CRITICAL=$( echo "$releasedetails" | jq .staticCritical )
          HIGH=$( echo "$releasedetails" | jq .staticHigh )
          if [ $CRITICAL -gt 0 ]
          then
            echo "##[error]Application has critical Issues"          
            curl -H 'Content-Type: application/json' -d '{"text": "Application has critical Issues in FOD"}' ${{secrets.TEAMS_WEBHOOK}}          
            #exit 1
          elif [ $HIGH -gt 0 ]
          then
            echo "##[error]Application has High Issues"
            curl -H 'Content-Type: application/json' -d '{"text": "Application has high Issues in FOD"}' ${{secrets.TEAMS_WEBHOOK}}
            #exit 1
          else
            echo "PASSED"
          fi
        continue-on-error: true

    Note: To enforce security gating in block mode please uncomment exit code 1.

    Jenkins¶

    Step 1: Store GitHub username and personal access token in Jenkins credentials.​​​​​​​

    Step 2: Store API key, API secret in key vault credentials.

    Step 3: create a Jenkins pipeline and use the following code.

    stages {

    Note:

    1) If we are Scanning any code with "FOD Plugin", once the scan found any security vulnerabilities in code then it will automatically fail the CI/CD pipeline.

    2) As we are using FodUpload jar file to Scan the code instead of FOD plugin. There is no such process to fail the pipeline automatically once the scan found any security vulnerabilities in code.

    3) we need to use the custom code to fail the pipelines while using FodUpload jar to scan the code. Include lines from "55 to 63" to fail the CI/CD pipeline.

    4) $? is a special variable in shell that reads the exit status of the last command executed.

    Concourse CI:¶

    Step 1: Store GitHub Username, Personal Access Token and API_KEY and API_SECRET in Hashicorp Vault.

    Step 2: Create a Pipeline.yml with following code

    resources:
  - name: source-code
    type: git
    source:
      uri: <Github_Url>
      branch: <branch_name>
      username: ((Github.username))
      password: ((Github.personal_access_token))

jobs:
  - name: source-code
    public: false
    plan:
      - get: source-code
        trigger: true
      - task: test-using-FOD-SAST
        config:
          platform: linux
          image_resource:
            type: registry-image
            source: { repository: ubuntu, tag: "latest" }
          params:
            FOD_API_KEY: ((API_CRED.FOD_API_KEY))
            FOD_API_SECRET: ((API_CRED.FOD_API_SECRET))
          inputs:
            - name: source-code
          run:
            path: /bin/sh
            args:
              - -c
              - |
                apt-get update
                echo "--------install java------"               
                apt-get update && DEBIAN_FRONTEND="noninteractive" TZ="Asia/india" apt-get install -y tzdata
                apt-get -y install openjdk-11-jre               
                echo "-------install zip----"
                apt-get -y install zip
                echo "-----install wget------"
                apt-get -y install wget
                echo "java --version"
                cd source-code
                zip -r uploadsourc.zip . -i /*.*
                wget -c https://github.com/fod-dev/fod-uploader-java/releases/download/v5.4.0/FodUpload.jar
                java -jar FodUpload.jar -ac ((API_CRED.FOD_API_KEY)) ((API_CRED.FOD_API_SECRET)) -rid 6560 -purl https://emea.fortify.com/ -                      apiurl https://api.emea.fortify.com/ -z uploadsourc.zip -ep 2 -rp 0 -pp 1 -apf -I 1

    Note:

    Use the variable name 'API_CRED.FOD_API_KEY' to pass fortify on demand API token in the pipeline. All the variable fields are configured in Hashicorp vault for reusability and manage the secrets ((Hashicorp vault variable)). GitHub and Fortify on demand details to be updated in the Hashicorp Vault. (Don't hard code username and password values in the pipeline) For steps to create a secret keys in HC Vault, refer here.
